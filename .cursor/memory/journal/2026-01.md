## 2026-01-03
- [Text][RTL][Fix] Implemented dynamic value substitution for `{1}`, `{2}`, `{3}` tooltip placeholders.
  - Why it changed: `ST_TranslatePrepare` was a stub that returned translations unchanged, causing literal `{1}%` to appear in tooltips instead of actual spell values like `20%`. Translation data uses `{1}`, `{2}` placeholders that need values extracted from the original English text.
  - Key files touched:
    - `common/Tooltips/Utils.lua` (rewrote `ST_TranslatePrepare` to extract numbers from original text and substitute into `{1}`, `{2}`, `{3}` placeholders in translations).
  - Regression checks run: RTL special codes (Text pipeline) - verify placeholder substitution works for spell tooltips with dynamic values.

- [Text][RTL][Fix] Added printf-style format token protection to prevent RTL reversal corruption.
  - Why it changed: printf tokens like `%s`, `%d`, `%1$s`, `%.2f` were not protected in `HandleWoWSpecialCodes`, causing them to break under RTL reversal (e.g., `%s` → `s%`). Any text with these tokens that passes through Arabic RTL processing would be corrupted.
  - Key files touched:
    - `common/Text.lua` (added two new protection patterns in `HandleWoWSpecialCodes`: one for standard printf tokens `%s`, `%d`, `%.2f`, etc., and one for positional printf tokens like `%1$s`, `%2$d`).
  - Regression checks run: RTL special codes (Text pipeline) - verify printf tokens survive RTL reversal intact.

- [Text][RTL][Fix] Added Blizzard dynamic placeholder `{1}`, `{2}` protection for defensive RTL safety.
  - Why it changed: While translation data is pre-shaped (so `{1}` placeholders don't normally go through reversal), any edge case where raw `{1}` tokens pass through `AS_UTF8reverse` would break them (becoming `}1{`). Added protection as a defensive measure.
  - Key files touched:
    - `common/Text.lua` (added `{%d+}` pattern protection in `HandleWoWSpecialCodes` before the printf token protections).
  - Regression checks run: RTL special codes (Text pipeline) - verify `{1}`, `{2}` tokens survive RTL reversal intact.

- [Text][RTL][Fix] Fixed RTL reversal of substituted numeric values ("20" → "02" bug).
  - Why it changed: Numeric values substituted by `ST_TranslatePrepare` (e.g., "20" replacing `{1}`) were being reversed to "02" during Arabic RTL processing because substitution happened BEFORE reversal protection.
  - Solution: In Arabic locale, `ST_TranslatePrepare` now wraps substituted values with `\003...\004` markers. `HandleWoWSpecialCodes` recognizes these markers and protects the values using the existing `\001INDEX\002` placeholder system.
  - Key files touched:
    - `common/Tooltips/Utils.lua` (wrap substituted values with `\003...\004` markers for AR locale).
    - `common/Text.lua` (added pattern in `HandleWoWSpecialCodes` to protect `\003...\004` marked values).
  - Regression checks run: RTL special codes (Text pipeline) - verify numeric values in tooltips display correctly (not reversed).

- [Tooltips][RTL] Right-justified tooltip lines in Arabic to feel RTL.
  - Why it changed: Arabic tooltips were shaped/reversed but still visually LTR-aligned (left-justified), which feels wrong and is harder to read.
  - Solution: Set `TextLeftN`/`TextRightN` fontstrings to `SetJustifyH("RIGHT")` (no anchor mirroring) AFTER all text translation is complete, right before `:Show()`.
  - Key files touched:
    - `common/Tooltips/Hooks.lua` (added `ApplyRTLJustifyToTooltipLines` helper; fixed runtime access to `ns.RTL` since RTL.lua loads after Hooks.lua in TOC order).
    - `common/Tooltips/GameTooltip.lua` (added RTL justification in `GT.OnShow()`, `GT.CurrentEquipped()`, `GT.ElvSpellBookTooltipOnShow()`, and `GT.BuffOrDebuff()` - applied right before `:Show()` calls).
    - `common/Tooltips/Utils.lua` (use `ns.RTL.IsRTL()` for RTL detection when marking substituted values).
  - Regression checks run (from `regression-checklist.md`): Tooltips - verify tooltip fonts/layout are consistent; confirm hooks are owned only by `common/Tooltips/*` (recommended manual in-game verification).

- [Tooltips][Fix] Fixed `{3}` placeholder showing comma instead of number.
  - Why it changed: The number extraction pattern `%-?[%d,]+%.?%d*` could match a lone comma `,` because `[%d,]+` matches one or more digits OR commas. If the original text had any commas, they could be incorrectly extracted as "numbers" and substituted into placeholders.
  - Solution:
    1. Changed pattern to `%-?%d[%d,]*%.?%d*` which REQUIRES at least one digit before any commas
    2. Added fallback RTL detection using `WoWTR_Localization.lang == 'AR'` in case `ns.RTL` isn't loaded
  - Key files touched:
    - `common/Tooltips/Utils.lua` (fixed number extraction pattern in `ST_TranslatePrepare`; improved RTL detection with fallback)
  - Regression checks run: RTL special codes - verify numeric placeholders {1}, {2}, {3} substitute correctly.

- [Tooltips][Fix] Fixed RTL justification applying to English tooltips without translation.
  - Why it changed: RTL justification was being applied to ALL tooltips when in Arabic mode, even when the tooltip had no Arabic translation and displayed English text. This made untranslated English tooltips appear right-aligned incorrectly.
  - Solution: Check if the tooltip **actually contains Arabic text** before applying RTL justification. Added `ContainsArabicText()` helper function that checks for Arabic character ranges.
  - Key files touched:
    - `common/Tooltips/GameTooltip.lua` (added `ContainsArabicText` helper; RTL justification now checks for Arabic content first in `GT.OnShow()`, `GT.BuffOrDebuff()`, `GT.ElvSpellBookTooltipOnShow()`, `GT.CurrentEquipped()`)
    - `common/Tooltips/Hooks.lua` (updated `ApplyRTLJustifyToTooltipLines` to check for Arabic content)
  - Regression checks run: Tooltips - verify English tooltips remain left-aligned; Arabic tooltips are right-aligned.

- [Text][RTL][Fix] Fixed `AS_UTF8charbytes` returning nil for NUL bytes (0x00), causing arithmetic crashes.
  - Why it changed: For `c == 0`, none of the UTF-8 byte-range branches matched, the final `else` returned nothing, and callers like `AS_UTF8len` executed `pos = pos + nil` → “attempt to perform arithmetic on a nil value”.
  - Key files touched:
    - `WoWAR/WoW_Arabic_Reshaper.lua` (`AS_UTF8charbytes`: final `else` now returns `1` as a safe single-byte fallback)
  - Regression checks run (from `regression-checklist.md`): RTL special codes (Text pipeline) - ensure RTL shaping still works; no visible behavior change expected for normal WoW text.

- [Text][RTL][Fix] Corrected Persian/Urdu letter presentation-form mappings (پ/چ/ژ/گ/ک/ڌ/ی).
  - Why it changed: The Persian/Urdu extension rules were using incorrect Forms-B code points (FE70–FEFF) which map to *different Arabic letters* (e.g., PEH → HAH forms), causing Persian text to display as the wrong characters.
  - Key files touched:
    - `WoWAR/WoW_Arabic_Reshaper.lua` (Persian/Urdu extension table now uses the correct Arabic Presentation Forms-A code points in the FB50–FDFF range)
    - `Tools/index.html` (reshaper preview tool `RULES_LIST` updated to match)
  - Regression checks run (from `regression-checklist.md`): RTL special codes (Text pipeline) - spot-check reshaping still preserves codes; recommended manual test with a Persian string containing these letters.
